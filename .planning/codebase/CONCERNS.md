## 1. Tech Debt

**Lines:** 853, 873, 942, 962, 1037, 1055, 1124, 1142, 1214, 1233  

The `Platform` class has repeated commented-out TODO blocks for a `CollectionDelta` change-notification mechanism across all major data setters (`getFileSprayWorkunits`, `getDataQuerySets`, `getLogicalFiles`, `getClusters`):

## 2. Known Bugs / Server Issues

### 2.1 ESP DFUGetDataColumns Does Not Return DATASET-Type Columns
**Source:** `WsDFU_getFileDataColumnsTestGeneration/WsDFU.getFileDataColumnsFinalReport_2025-11-11.md`  
**Tests:** `testGetFileDataColumns_AllTypes` (CFT-003), `testGetFileDataColumns_NestedColumns` (CFT-007)  
**Status:** `@Category(UnverifiedServerIssues.class)` — 2 permanently failing tests  

The ESP `DFUGetDataColumns` service excludes nested DATASET-type column fields from its response. A file with a `childDataset` field returns 20 columns instead of the expected 21. Workaround: use `DFUInfo` service + `EclRecordReader` for schema retrieval when nested structures are required. This is a server limitation, not a client bug.

### 2.2 WsResources Test Generator: All 53 Tests Failed Due to Method Name Mismatch
**Source:** `scripts/wsclient-test-generator/WsResources_FullServiceTestGeneration_2026-03-04/WsResources.AllMethodsBatchAnalysis_Iteration1_2026-03-04.md`  

In iteration 1, all 53 generated `WsResourcesClientTest` tests reported "No tests were executed!" because the generator produced method names like `test001Ping` while the test runner expected `ping`. The test file was regenerated in iteration 2. This demonstrates a structural fragility in the test generator's method naming convention — whenever the generator is re-run, the naming contract must be manually verified.

### 2.3 getFileDataColumns ClassCastException (Now Fixed, Risk Pattern Remains)
**Source:** `WsDFU_getFileDataColumnsTestGeneration/WsDFU.getFileDataColumnsFinalReport_2025-11-11.md`, Iteration 1  
**File:** `wsclient/src/main/java/org/hpccsystems/ws/client/HPCCWsDFUClient.java`  

A `ClassCastException` was thrown when `getFileDataColumns()` attempted a direct array cast of `ArrayOfDFUDataColumn` — an Axis2 wrapper class — to `DFUDataColumn[]`. Fixed by adding an `instanceof` check and calling `.getDFUDataColumn()`. This same pattern (directly casting Axis2-returned wrapper objects) is likely present in other methods across `HPCCWsDFUClient` and other WS clients that haven't yet been exercised by tests.

### 2.4 Error Message Assertion Mismatch: "Cannot find" vs. "Cannot resolve"
**Source:** `WsDFU_getFileDataColumnsTestGeneration/WsDFU.getFileDataColumnsFinalReport_2025-11-11.md`, Iteration 1  
**Test:** `testGetFileDataColumns_FileNotFound`  

The actual ESP error message uses "Cannot resolve" but code comments and older tests expected "Cannot find". Test was fixed, but any other code that parses or matches ESP error message text is fragile to server-side message wording changes.

## 3. Fragile Areas

### 3.1 Multi-Version WsWorkunits Stub Dispatch (WsWorkunitsClientStubWrapper)
**File:** `wsclient/src/main/java/org/hpccsystems/ws/client/wrappers/wsworkunits/WsWorkunitsClientStubWrapper.java`  

The codebase maintains four separate versioned WsWorkunits stubs (`v1_58`, `v1_62`, `v1_69`, `latest`). `WsWorkunitsClientStubWrapper` selects the appropriate stub at runtime based on `targetHPCCBuildVersion`. This runtime version dispatch logic is not tested for edge cases around version boundaries. If the HPCC platform version string format changes, the wrong stub may be selected silently.

### 3.2 `HPCCWsWorkUnitsClient` Uses Both `stubWrapper` and Direct `stub` Cast
**File:** `wsclient/src/main/java/org/hpccsystems/ws/client/HPCCWsWorkUnitsClient.java`  
**Lines:** 276–277, 316, 741, 773, 1027  

The client simultaneously assigns both a versioned `stubWrapper` and a `stub` from the base class, then uses both in different methods:
```java
stubWrapper = new WsWorkunitsClientStubWrapper(...);
stub = stubWrapper.getLatestStub();
stub = setStubOptions(new WsWorkunitsStub(...), conn);  // Overwrites!
```
`stub` is immediately overwritten by a fresh `WsWorkunitsStub` instance that bypasses the versioned stub selection. Methods that cast `stub` to `WsWorkunits` (lines 316, 741, 773, 1027) may call the wrong version. This dual-path initialization is highly confusing and likely a latent bug.

### 3.3 BaseHPCCWsClient `isComplete()` Always Returns `false`
**File:** `wsclient/src/main/java/org/hpccsystems/ws/client/BaseHPCCWsClient.java`  
**Line:** 617  

Because `isComplete()` is a stub that always returns `false`, the `DataSingleton` framework's cache-validity logic is permanently broken for all WS clients. Any code that conditions behavior on `isComplete()` will always see a stale/incomplete state.

### 3.4 Platform Observer Pattern: Data Updates Have No Notification
**File:** `wsclient/src/main/java/org/hpccsystems/ws/client/platform/Platform.java`  

The `Platform` class holds mutable collections (`fileSprayWorkunits`, `dataQuerySets`, `logicalFiles`, `clusters`) that are refreshed on demand. Without functional `notifyObservers()` calls (all commented out as TODO), UI or integration consumers cannot react to data changes. Any consumer polling `Platform` for data changes must re-check every field manually.

### 3.5 Reflection-Based Access to Axis2 Classes in Tests
**Source:** `WsResources.pingAnalysis_2026-03-04.md`, test `pingWithNullStubTest`; `WsResources.webLinksAnalysis_2026-03-04.md`, test `testWebLinksConfiguredWebLinkResourceType`  

Some tests use reflection to access private `stub` fields or check for the presence of methods like `getResourceType()` on raw Axis2 classes. These tests will silently skip when the method is absent (older WSDL), but may silently test nothing useful after WSDL regeneration changes method signatures.

### 3.6 WsResources Tests Are Skipped on Bare-Metal Clusters
**Source:** `scripts/wsclient-test-generator/WsResources_FullServiceTestGeneration_2026-03-04/WsResources.targetQueryAnalysis_2026-03-04.md`  

Multiple `WsResources` tests guard against non-containerized deployments with `Assume.assumeTrue(isContainerizedTarget)`. On bare-metal CI, a significant fraction of the test suite is effectively skipped. There is no bare-metal equivalent test coverage for containerized-only code paths.

## 4. Dependency Risks

### 4.1 Spark Version: Default Is End-of-Life Spark 2.4.6 / Scala 2.11
**File:** `spark-hpcc/pom.xml`  
```xml
<spark.runtime.version>2.4.6</spark.runtime.version>
<scala.binary.version>2.11</scala.binary.version>
```
The **default** Spark build is 2.4.6, released in 2019 and unsupported since January 2023. Scala 2.11 reached end-of-life in 2022. This default is used when no build profile is explicitly activated. Users who run `mvn install` without `-Pspark33` or `-Pspark34` get an EOL Spark artifact.

### 4.2 JSch 0.1.54 — Ancient and Unmaintained (2016)
**File:** `pom.xml`, line 53  
```xml
<jsch.version>0.1.54</jsch.version>
```
JSch 0.1.54 was released in 2016. The `com.jcraft:jsch` library has been effectively abandoned; the community-maintained fork `com.github.mwiede:jsch` is the modern replacement. Version 0.1.54 has known issues with modern OpenSSH key formats (Ed25519, ECDSA) and is incompatible with OpenSSH ≥ 8.8 which disabled RSA-SHA1 by default. This affects `Sftp.java` and any secure file transfer functionality.

### 4.3 Maven Surefire Plugin 2.22.1 — Outdated
**File:** `pom.xml`  
```xml
<maven.surefire.version>2.22.1</maven.surefire.version>
```
Surefire 2.22.1 is from 2018. Current stable is 3.x. Known issues include JUnit 5 limitations, inconsistent test isolation, and the confirmed behavior seen in the WsResources test run where `failIfNoTests` semantics caused all tests to report as `BUILD FAILURE` even though the tests themselves were valid. Upgrading to 3.x would also improve parallel test execution support.

### 4.4 javax.mail 1.4 — Jakarta Migration Not Done
**File:** `pom.xml`  
```xml
<javax.mail.version>1.4</javax.mail.version>
```
`javax.mail:mail:1.4` was released in 2009. Modern Java EE / Jakarta EE projects have migrated to `jakarta.mail`. Version 1.4 predates many security fixes and modern TLS/SSL improvements. No evidence that email functionality in the library is actively tested.

### 4.5 Maven GPG Plugin 1.6 — End-of-Life
**File:** `pom.xml`  
```xml
<maven.gpg.version>1.6</maven.gpg.version>
```
Plugin version 1.6 is the last 1.x release; it predates support for GPG 2.1+ agent-based signing. On macOS with GPG 2.x (default via Homebrew), this version frequently causes signing failures during `mvn deploy -P release`.

### 4.6 Maven Source Plugin 2.2.1 — EOL
**File:** `pom.xml`  
```xml
<maven.source.version>2.2.1</maven.source.version>
```
Released in 2013. Current stable is 3.x. No functional blocker but creates an inconsistency with the build tooling age.

### 4.7 Maven Deploy Plugin 2.8.2 and Install Plugin 2.5.2 — EOL
Both are over a decade old. The `central-publishing-maven-plugin` (`0.7.0`) used for Maven Central publication is modern, but these supporting plugins create a mixed-vintage build toolchain.

### 4.8 OpenTelemetry Semconv at Alpha Version
**File:** `pom.xml`  
```xml
<opentelemetry.semconv.version>1.25.0-alpha</opentelemetry.semconv.version>
```
An alpha dependency in production library code is a stability risk. Semantic convention attributes may change between alpha releases, breaking span attribute contracts without a major version bump.

### 4.9 log4j Pinned in spark-hpcc to 2.17.1 (Type Scope Concern)
**File:** `spark-hpcc/pom.xml`  
```xml
<dependency>
    <groupId>org.apache.logging.log4j</groupId>
    <artifactId>log4j-core</artifactId>
    <version>2.17.1</version>
    <type>pom</type>
</dependency>
```
Declaring `log4j-core` with `<type>pom</type>` means it acts as a BOM import, not a compile/runtime dependency injection. The actual log4j version used at runtime will be whatever Spark provides. The README still calls out the log4j CVE vulnerability, suggesting the mitigation may be incomplete.

## 5. Version Compatibility

### 5.1 Spark Version Matrix: Three Active Profiles, Default Is EOL
**File:** `spark-hpcc/pom.xml`  
```
spark24: Spark 2.4.6 / Scala 2.11 (DEFAULT — EOL)
spark33: Spark 3.3.2 / Scala 2.12
spark34: Spark 3.4.1 / Scala 2.12
```
Spark 3.5.x exists but has no corresponding profile. Spark 3.3.x will reach EOL. The Databricks Runtime is not covered. There is no CI matrix testing all three (or more) profiles simultaneously.

### 5.2 HPCC Platform Minimum Version Raised in 10.0.0
**File:** `MIGRATION-10.0.md`  

HPCC4J 10.0 removed support for HPCC Platform v7.0.0 exactly (wsdfu v1_39 API). The minimum is now v7.0.x where x > 0 (effectively v7.0.2+). Three methods were removed or replaced with `UnsupportedOperationException`. Any downstream library that pinned to 9.x and consumed the removed `SecAccessType` overloads of `getFileAccessBlob()` or `getFileAccess()` will break on upgrade.

### 5.3 WsResources targetQuery Requires Service Version ≥ 1.03; webLinks ≥ 1.01
**Source:** `WsResources.ServiceAnalysis_2026-03-04.md`, Section D  

No version guard is enforced at the client level for `targetQuery` and `webLinks`. On older HPCC clusters, calling these methods will produce a server error, which surfaces as a generic exception rather than a meaningful "unsupported version" message.

### 5.4 Java 8 as Minimum Baseline
**File:** `pom.xml`  
```xml
<maven.compiler.release>8</maven.compiler.release>
```
Java 8 reached Oracle end-of-support in 2019. Many organizations are at Java 17 LTS or Java 21 LTS. Maintaining Java 8 compatibility constrains API design (no records, no sealed classes, limited lambda improvements) and prevents adoption of modern dependency versions that have dropped Java 8 support.

## 6. Generated Code Risks

### 6.1 Axis2 Stub Generation: Unchecked / Unsafe Operations at Compile Time
**Source:** `WsResources.AllMethodsFailureReport_Iteration1_2026-03-04.md` (compilation warnings)  
**File:** `wsclient/src/main/java/org/hpccsystems/ws/client/gen/axis2/wsworkunits/v1_56/WsWorkunitsStub.java`  

The generated `WsWorkunitsStub.java` emits unchecked/unsafe operation warnings at compile time:
```
[WARNING] Some input files use unchecked or unsafe operations.
[WARNING] Recompile with -Xlint:deprecation for details.
```
Generated stubs use raw types throughout. These cannot be fixed without regenerating (which changes ~millions of lines) or suppressing all warnings.

### 6.2 Generated Stubs Compiled at Deprecation Warning Level
Multiple generated stub classes use internal Axis2 APIs now marked deprecated. The build does not treat deprecation warnings as errors (`-Werror`), so these silently accumulate. Any future Axis2 major version upgrade could break the generated code entirely.

### 6.3 Direct Cast to Axis2-Generated Wrapper Types Without instanceof Guard
**File:** `wsclient/src/main/java/org/hpccsystems/ws/client/HPCCWsDFUClient.java` (post-fix evidence)  

The `getFileDataColumns()` `ClassCastException` (see §2.3) was caused by directly casting an Axis2 response to an inner array type without checking for the outer wrapper class first. The test generator discovered this pattern; the same risk exists wherever Axis2 array-of-X return types are consumed. No audit has been done across all methods for this anti-pattern.

### 6.4 WSDL Regeneration Workflow Is a Manual, Error-Prone Process
**File:** `wsclient/utils/wsInterfaceUpdater.py`, `wsclient/utils/README.md`  

Stub regeneration requires running a Python script that invokes external WSDL/ESDL tooling, then manually reviewing diffs to ensure wrapper classes are regenerated correctly. There is no automated check that wrapper classes are in sync with generated stubs. If an engineer regenerates stubs without regenerating wrappers (or vice versa), the discrepancy will only surface at runtime via reflection or cast failures.

### 6.5 ANTLR generated EclRecordLexer/EclRecordParser Have Deprecated Members
**Files:**  
- `wsclient/src/main/java/org/hpccsystems/ws/client/antlr/gen/EclRecordLexer.java` lines 70, 87  
- `wsclient/src/main/java/org/hpccsystems/ws/client/antlr/gen/EclRecordParser.java` lines 75, 92  

Generated ANTLR classes contain `@Deprecated` methods (likely from ANTLR's own runtime). These should not be modified directly; they must be regenerated when the grammar file changes. The grammar file version is not pinned in `pom.xml` comments, so ANTLR version drift could silently change grammar behavior.

## 8. Security Considerations

### 8.1 Credentials Stored as Plain `String` in `Connection` Object
**File:** `wsclient/src/main/java/org/hpccsystems/ws/client/utils/Connection.java`  
**Line:** 43  
```java
private String password = null;
```
Passwords are held in heap-resident `String` objects, which are GC-managed and may persist in memory longer than necessary. Java security best practices recommend `char[]` for password storage so it can be explicitly zeroed. The `Connection` object is passed into object pools and cached, extending the credential lifetime further.

### 8.2 Basic Auth Credentials Encoded in HTTP Header (Not HTTPS-Enforced)
**File:** `wsclient/src/main/java/org/hpccsystems/ws/client/utils/Connection.java`  
**Line:** 124  
```java
return new String(encoder.encode((userName + ":" + password).getBytes()));
```
Credentials are Base64-encoded for HTTP Basic Auth. There is no enforcement that connections use HTTPS before transmitting credentials. Callers can create a `Connection` with `http://` and send plaintext-equivalent credentials over the wire.

### 8.3 JSch 0.1.54 Incompatible with Modern SSH Security Standards
**File:** `wsclient/src/main/java/org/hpccsystems/ws/client/utils/Sftp.java`  

JSch 0.1.54 (2016) predates support for OpenSSH's post-2021 security defaults (removal of `ssh-rsa` as a default algorithm, rejection of SHA1-based signatures). SFTP operations to modern Linux servers will silently fail or fall back to weak cipher suites. This is a security regression risk, not just a compatibility issue.

### 8.4 Apache Spark Standalone Mode Security Warning
**File:** `spark-hpcc/README.md`  

The README documents a known Apache Spark vulnerability: the standalone resource manager's master node can execute arbitrary code from specially crafted requests if authentication is not enabled (`spark.authenticate`). The library ships with no enforcement or documentation that cluster authentication must be enabled when using `spark-hpcc`.

### 8.5 Passwords Passed as SFTP Method Arguments (Visible in Stack Traces)
**File:** `wsclient/src/main/java/org/hpccsystems/ws/client/utils/Sftp.java`  
**Line:** 20  
```java
static public void lzPut(..., String password, ...)
```
Passwords are passed as plain `String` method arguments, making them visible in stack trace dumps, thread dumps, and any APM tooling that captures method arguments.

### 8.6 Remote Source Passwords in `CopyWrapper` and `AddRemoteRequestWrapper`
**Files:**  
- `wsclient/src/main/java/org/hpccsystems/ws/client/wrappers/gen/filespray/CopyWrapper.java`  
- `wsclient/src/main/java/org/hpccsystems/ws/client/wrappers/gen/wsdfu/AddRemoteRequestWrapper.java`  

`srcpassword` fields are plain `String` members of `CopyWrapper` and `AddRemoteRequestWrapper`. These objects are serialized into SOAP payloads. If TLS is not configured, source cluster passwords are transmitted in cleartext.

## 9. Performance Concerns

### 9.1 `Platform.java` Data Methods Are Not Thread-Safe for Concurrent Refresh
**File:** `wsclient/src/main/java/org/hpccsystems/ws/client/platform/Platform.java`  

Multiple `Platform` methods that refresh internal collections (`getFileSprayWorkunits`, `getDataQuerySets`, `getLogicalFiles`, `getClusters`) are `synchronized` at the method level. However, the combination of checking a stale-flag and then fetching is not atomic. Concurrent callers may trigger redundant refreshes or observe partially-updated collection states.

### 9.2 `HPCCWsClientPool` Expiry Check Not Benchmarked
**File:** `wsclient/src/main/java/org/hpccsystems/ws/client/HPCCWsClientPool.java`  

The pool uses a timeout (`DEFAULT_EXPIRE_MILLIS`) but there is no documented capacity limit or eviction policy beyond expiry. Under high-concurrency workloads, pool starvation or unbounded growth has not been analyzed or tested.

### 9.3 Spark Write Path: No Documented Performance Benchmarks
**Directory:** `spark-hpcc/src/main/`  

`HpccFileWriter` handles distributed write partitioning. There are no documented performance benchmarks, and no known test asserts that the partition-to-node mapping scales correctly with large datasets. `RowServiceOutputStream` contains a deprecated write method, suggesting the current API path has not been exhaustiveness-tested under load.

### 9.4 ECL Record Parsing via ANTLR on Every Schema Lookup
**File:** `wsclient/src/main/java/org/hpccsystems/ws/client/antlr/`  

ECL record definitions are parsed by an ANTLR grammar (`EclRecordParser`) at runtime. There is no caching layer between `DFUInfo` responses and the ANTLR parse step. Repeated schema queries for the same file (common in Spark partition enumeration) rebuild the parse tree each time.

## 10. Maintenance Burden

### 10.1 WsWorkunits Has Four Versioned Stubs to Maintain in Parallel
**Directory:** `wsclient/src/main/java/org/hpccsystems/ws/client/wrappers/wsworkunits/`  

Every new HPCC platform feature that extends WsWorkunits requires evaluating which of the four stub versions (`v1_58`, `v1_62`, `v1_69`, `latest`) need updating. The `WsWorkunitsClientStubWrapper` version dispatch must also be updated. This branching maintenance cost grows with each platform release.

### 10.2 Wrapper Code Generation Script Requires Python + External ESDL/WSDL Tooling
**File:** `wsclient/utils/wsInterfaceUpdater.py`  

The wrapper generation script has its own dependency chain (Python, ESDL tooling) that is not documented in the root `README.md`. Contributors without the correct environment cannot regenerate wrappers, creating a high barrier to entry for adding new ESP service support.

### 10.3 Test Generator (TestGeneratorAgent.py) Is a Full Agentic Workflow
**File:** `scripts/wsclient-test-generator/TestGeneratorAgent.py`  

The AI-powered test generator requires iterative execution with manual review between iterations. As seen in the WsResources case, initial test generation produced 10 methods out of 53 needed, requiring a complete iteration to regenerate. This is a multi-hour process per service. With 10+ ESP clients in the library, full test coverage generation is a significant ongoing investment.

### 10.4 Eclipse Formatter Required but Not Enforced in Build
**File:** `eclipse/HPCC-JAVA-Formatter.xml`  

The HPCC-JAVA-Formatter.xml is the mandated code style, but there is no `checkstyle` or `spotless` Maven plugin configured to enforce it. Code formatting divergence is detected only during code review, creating maintenance friction.

### 10.5 spark-hpcc README Still References Log4j CVE as an Active Concern
**File:** `spark-hpcc/README.md`  

The README has a prominently placed security table warning about the Log4Shell vulnerability (log4j < 2.15.0). This version range was fully superseded years ago. Leaving this warning prominently displayed creates false alarm for new developers and suggests the README is not maintained alongside releases.

### 10.6 dfsclient Kubernetes Configuration Documented Externally Only
**File:** `dfsclient/README.md`  

Using `dfsclient` with containerized HPCC (Kubernetes) requires additional local configuration steps documented exclusively on an external GitHub Wiki page. If the Wiki is lost or not updated, this critical configuration knowledge is unavailable. No in-repo documentation exists for the Kubernetes workflow.
